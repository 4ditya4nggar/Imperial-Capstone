# Model Card: GP-Based Weekly Black-Box Query Proposer
# Overview

Model name: GP + Expected Improvement Weekly Proposer
Implementation: black_box_optimizer.py
Execution notebook: notebooks/bboWeeklyRunner.ipynb

This model supports sequential optimisation in black-box settings where the underlying function is unknown. It uses a Gaussian Process (GP) surrogate model combined with the Expected Improvement (EI) acquisition function to recommend the next candidate query point based on previously observed data.

The optimiser analyses existing input–output observations and proposes a new candidate point (x_next). The current implementation only generates recommendations and does not directly evaluate the external objective function.

# Intended Use

The model is designed for optimisation tasks with the following characteristics:

The objective function is unknown and treated as a black-box

Evaluations are expensive, so only a small number of observations are available

The search space is continuous and bounded within [0,1]

Queries are generated sequentially, typically once per week

In this repository, the optimiser is applied to eight benchmark functions with dimensionalities ranging from 2 to 8 variables. Initial observations are stored in data/initial_data/, while weekly query records are maintained under queries/week*/.

# Model Details

# Surrogate Model

The optimisation relies on a GaussianProcessRegressor from scikit-learn with the following configuration:

Kernel: Matérn (ν = 2.5)

Noise parameter: alpha = 1e-6

Output normalisation enabled

The GP surrogate estimates both the predicted function value and its uncertainty, enabling acquisition-based decision making.

# Acquisition Function

The model selects candidate points using Expected Improvement (EI) with an exploration parameter ξ = 0.01. EI balances the trade-off between exploring uncertain regions and exploiting areas predicted to yield better results.

# Candidate Generation

Candidate points are generated by random sampling within the bounded domain [0,1]^d.
A pool of 5000 candidates is evaluated, and the point with the highest EI score is selected. The chosen point is clipped to the domain limits and rounded to six decimal places.

## Inputs and Outputs

# Inputs

initial_inputs: array of previously evaluated points (n_samples, n_features)

initial_outputs: observed objective values (n_samples,)

max_queries: number of optimisation iterations

seed: random seed for reproducibility

# Outputs

The optimiser returns the input and output arrays and prints the proposed candidate (x_next) for each iteration. Objective evaluation must be performed externally.

## Assumptions and Limitations

# Assumptions

The objective function is reasonably smooth for GP modelling

The existing observations provide enough information to guide EI ranking

The optimisation domain is continuous and bounded

# Limitations

The optimiser does not evaluate the objective function internally

New observations must be manually appended to the dataset

Random candidate search may miss optimal points in higher-dimensional problems

Fixed GP hyperparameters may not suit all functions

## Evaluation Context

The model is evaluated across the eight capstone benchmark functions using sequential weekly optimisation rounds. Performance is assessed by observing improvements in the best objective value and the stability of suggested query points as more data are collected.

# Transparency and Reproducibility

All optimisation logic is implemented in black_box_optimizer.py, and the weekly workflow can be reproduced using the provided notebook runner. Query history and optimisation artefacts are stored in clearly structured directories to support traceability.
